      ### 1. better_sqlite3_load
        Symptom: `[fail] better_sqlite3_load — failed to load: ...` in doctor output.
        Likely cause: Native module prebuilt binaries missing for your Node version; node-gyp compile failed.
        Fix:
          - Verify Node 22.x: `node --version`
          - Reinstall: `rm -rf node_modules && npm install`
          - If that fails on a Linux box: install build deps (`apt install build-essential python3` or equivalent) then `npm rebuild better-sqlite3`.
        See also: [agent_docs/decisions/0006-fixture-only-tests.md](../../agent_docs/decisions/0006-fixture-only-tests.md), [INSTALL.md](../../INSTALL.md) Prerequisites.

      ### 2. napi_keyring_load
        Symptom: `[fail] napi_keyring_load — failed to load: ...`.
        Likely cause: On Linux, libsecret-1-dev missing. On macOS, native module compile failed.
        Fix:
          - macOS: `npm rebuild @napi-rs/keyring`.
          - Linux: `sudo apt install libsecret-1-dev` then reinstall; OR set `RECOVERY_LEDGER_FORCE_FILE_STORE=1` to use the chmod 600 fallback per [agent_docs/decisions/0002-single-flight-oauth-refresh.md](../../agent_docs/decisions/0002-single-flight-oauth-refresh.md).
        See also: agent_docs/decisions/0002-single-flight-oauth-refresh.md.

      ### 3. mcp_stdout_purity
        Symptom: `[fail] mcp_stdout_purity — non-JSON-RPC bytes detected on stdout`.
        Likely cause: A code path is writing to `process.stdout` outside the allowed CLI doctor command path. Probably a stray `console.log` or `process.stdout.write` introduced in a recent change.
        Fix:
          - Run `bash scripts/ci-grep-gates.sh` and inspect Gate B (direct stdout writes outside src/cli/**) + Gate C (process.stdout.write outside src/cli/commands/**). Both should be green; a fail means a regression.
          - Rebuild: `npm run build` then re-run `recovery-ledger doctor`.
        See also: [agent_docs/decisions/0001-mcp-stdout-purity.md](../../agent_docs/decisions/0001-mcp-stdout-purity.md).

      ### 4. auth
        Symptom: `[fail] auth — no tokens found at ~/.recovery-ledger/tokens.json (or keychain)`.
        Likely cause: You have not run `recovery-ledger auth` yet, or the tokens file was deleted/moved.
        Fix:
          - Run `recovery-ledger init` (if config is also missing) then `recovery-ledger auth`. Follow the browser flow.
          - On Linux without libsecret, the file fallback path is `~/.recovery-ledger/tokens.json` (chmod 600).
        See also: [docs/install/whoop-app.md](whoop-app.md), [INSTALL.md](../../INSTALL.md).

      ### 5. token_freshness
        Symptom: `[warn] token_freshness — expires in 12m` or `[fail] token_freshness — expired 2h ago`.
        Likely cause: Token is close to / past expiry; refresh has not run (sync hasn't been invoked recently).
        Fix:
          - `recovery-ledger sync` will trigger a refresh via the single-flight gate.
          - If refresh fails: `recovery-ledger auth` to re-authorize.
        See also: agent_docs/decisions/0002-single-flight-oauth-refresh.md.

      ### 6. whoop_roundtrip
        Symptom: `[fail] whoop_roundtrip — WHOOP returned 401 after refresh — run recovery-ledger auth` OR `[warn] WHOOP returned 403 — scopes may have drifted`.
        Likely cause: Token is invalid (re-auth required) OR the WHOOP dashboard scope list was edited and no longer matches what `init` registered.
        Fix:
          - 401: `recovery-ledger auth` to re-authorize.
          - 4xx other than 401: visit https://developer.whoop.com/dashboard/applications and verify the scope list matches [docs/install/whoop-app.md](whoop-app.md) Step 4.
          - To skip this check while debugging: `recovery-ledger doctor --offline`.
        See also: docs/install/whoop-app.md, [agent_docs/decisions/0007-whoop-read-only.md](../../agent_docs/decisions/0007-whoop-read-only.md).

      ### 7. db_open
        Symptom: `[fail] db_open — no DB handle injected` OR `[fail] db_open — pragma probe threw: ...`.
        Likely cause: Bootstrap failed to open the database (filesystem permission, disk full, corruption preventing the WAL pragma).
        Fix:
          - Verify `~/.recovery-ledger/recovery-ledger.sqlite` exists and is readable.
          - Check disk space: `df -h ~`.
          - If the file is corrupted: see `db_integrity` section below.
        See also: agent_docs/decisions/0006-fixture-only-tests.md.

      ### 8. db_integrity
        Symptom: `[fail] db_integrity — PRAGMA integrity_check returned N row(s); first: ...`.
        Likely cause: Database file is corrupt — power loss mid-write, disk failure, or external editing.
        Fix:
          - Restore from a pre-migration backup: `ls -1 ~/.recovery-ledger/backups/` shows available backups.
          - `cp ~/.recovery-ledger/backups/<latest>.sqlite ~/.recovery-ledger/recovery-ledger.sqlite`.
          - Re-run `recovery-ledger doctor` to confirm.
        See also: db_schema_version (related — both surface migration-state issues).

      ### 9. db_schema_version
        Symptom: `[fail] db_schema_version — schema at migration N/M — restore from <backupPath>: cp <backup> <dbFile>` OR `... — extra rows in __drizzle_migrations`.
        Likely cause: A migration was applied without its `.sql` file being shipped (downgrade scenario) OR a migration row was inserted without the SQL running (orphaned row per Phase 3 D-08).
        Fix:
          - If the doctor detail string includes a backup-path remediation: copy the backup over the current DB exactly as shown.
          - If no backup exists: file an issue — the migrator's fails-closed posture means manual recovery is required.
        See also: agent_docs/decisions/0006-fixture-only-tests.md.

      ### 10. db_wal_size
        Symptom: `[warn] db_wal_size — WAL <N>MB (>32MB; checkpoint is lagging)` OR `[fail] db_wal_size — WAL <N>MB exceeds journal_size_limit=64MB`.
        Likely cause: WAL has not been checkpointed; sync was interrupted before its final `wal_checkpoint(TRUNCATE)` ran (Phase 3 D-32).
        Fix:
          - Run `recovery-ledger sync` to completion (it ends with a TRUNCATE checkpoint).
          - If that fails: force a checkpoint via a tmp SQL session: `sqlite3 ~/.recovery-ledger/recovery-ledger.sqlite "PRAGMA wal_checkpoint(TRUNCATE);"`.
        See also: Phase 3 SYNC-06 + `src/infrastructure/db/connection.ts` D-30 pragma block.

      ### 11. last_sync_recency
        Symptom: `[warn] last_sync_recency — last sync 3d ago — run recovery-ledger sync (status: ok)` OR `[fail] last sync 10d ago — exceeds 7d threshold`.
        Likely cause: Sync has not run recently.
        Fix:
          - `recovery-ledger sync` to refresh.
          - Set up scheduled sync per [docs/install/launchd.md](launchd.md) so this doesn't recur.
        See also: docs/install/launchd.md.

      ### 12. most_recent_scored_day
        Symptom: `[warn] most_recent_scored_day — most recent SCORED day <date>, 3d ago — run recovery-ledger sync (cycles, recovery; sleep at <date>)`.
        Likely cause: Sync ran recently but data is all PENDING_SCORE (last night's cycle hasn't been scored by WHOOP yet) OR the SCORED data is stale.
        Fix:
          - Wait 12-24h for WHOOP to score recent data, then re-run.
          - If `last_sync_recency` is also failing, fix that first (see section above).
        See also: [agent_docs/decisions/0003-score-state-discipline.md](../../agent_docs/decisions/0003-score-state-discipline.md).

      ### 13. data_quality_counts
        Symptom: `[pass] data_quality_counts — cycles: 142 scored, 3 pending, ...` (this check is informational; it always reports pass).
        Likely cause: N/A — this is a visibility check, not a fail surface. Pitfall 19 in research (silent missing days) is what this exposes.
        Fix:
          - Use the counts to identify resources with high pending / unscorable counts.
          - High pending counts on recent days are expected (WHOOP takes time to score).
          - High unscorable counts may signal a hardware sensor issue worth checking in the WHOOP app.
        See also: agent_docs/decisions/0003-score-state-discipline.md.

      ### 14. concurrent_writers_stress
        Symptom: `[pass] concurrent_writers_stress — skipped — run with --stress to enable` by default. When enabled and failing: `[fail] concurrent_writers_stress — <W> of 4 workers failed: exit 1 (SQLITE_BUSY at iteration ...)`.
        Likely cause: SQLite `busy_timeout=5000` is being beaten — two writers are racing past the BEGIN IMMEDIATE lock.
        Fix:
          - This is a power-user diagnostic. If it fails, capture the doctor output and file an issue with the DB file size + your hardware specs.
          - Confirm no other process is writing to `~/.recovery-ledger/recovery-ledger.sqlite` (e.g., a stale sync child).
        See also: agent_docs/decisions/0002-single-flight-oauth-refresh.md (the single-flight gate design).

      Imports:
        import { readFileSync } from 'node:fs';
        import { resolve } from 'node:path';
        import { describe, expect, test } from 'vitest';
        import { CHECK_NAMES } from '../../src/services/doctor/checks/check-names.js';

      Body:
        describe('troubleshooting.md coverage', () => {
          const md = readFileSync(
            resolve(__dirname, '..', '..', 'docs', 'install', 'troubleshooting.md'),
            'utf8',
          );
          const h2s = [...md.matchAll(/^## (\S+)$/gm)].map((m) => m[1]);
          const expected = Object.values(CHECK_NAMES);

          test('every CHECK_NAMES value has a matching ## H2 in troubleshooting.md', () => {
            for (const name of expected) {
              expect(
                h2s,
                `missing troubleshooting section: ## ${name} — add a section to docs/install/troubleshooting.md per D-08 template`,
              ).toContain(name);
            }
          });

          test('H2 anchors appear in the same order as Object.values(CHECK_NAMES)', () => {
            // Filter h2s to only those that match a CHECK_NAMES value (ignore any other H2s like
            // 'Troubleshooting Recovery Ledger' which would be H1 anyway).
            const filtered = h2s.filter((h) => (expected as readonly string[]).includes(h));
            expect(
              filtered,
              'troubleshooting.md H2 order does not match CHECK_NAMES declaration order',
            ).toEqual(expected);
          });

          test('no EXTRA H2 anchors beyond CHECK_NAMES', () => {
            // Identify H2s that look like check names (lowercase + underscores) but aren't in CHECK_NAMES.
            const checkLike = h2s.filter((h) => /^[a-z_]+$/.test(h));
            const extras = checkLike.filter((h) => !(expected as readonly string[]).includes(h));
            expect(
              extras,
              `unexpected troubleshooting H2s (not in CHECK_NAMES): ${extras.join(', ')}`,
            ).toEqual([]);
          });
        });

    Why `(expected as readonly string[]).includes(h)`: TypeScript narrows `Object.values(CHECK_NAMES)` to the literal union type; the `includes` predicate widens to `string` for the runtime comparison. The cast is explicit and documented.

    Why the third test (no extras): prevents the stale-section case — a developer who renames a CHECK_NAMES value without updating the H2 leaves the old H2 orphaned. The third test catches this.
