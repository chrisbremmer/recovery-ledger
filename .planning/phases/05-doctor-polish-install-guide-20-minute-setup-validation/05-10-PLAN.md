Stopwatch boundary (D-12 — authoritative):
  pre-stopwatch: setup tmp dirs + env + MSW; clone-equivalent cp src → tmpRepo (NOT timed)
  STOPWATCH START
    npm install (dominant cost — native-module compile)
    npm run build
    recovery-ledger init (non-interactive via env-var creds)
    recovery-ledger auth (--no-browser or in-process; MSW token endpoint)
    recovery-ledger sync (MSW resource endpoints)
    recovery-ledger review daily
  STOPWATCH END
  assert elapsed < STOPWATCH_BUDGET_MS

Realism trade-off (D-14): MSW intercepts at the undici-fetch level IN-PROCESS only. Spawned children
DO NOT inherit MSW interception by default. Resolution: invoke init/auth/sync/review IN-PROCESS via
direct import of the command functions (after npm install + npm run build are done as subprocesses
so the install + build cost IS counted). This preserves D-12's "user-felt friction" goal because
npm install + build dominate the runtime budget; in-process command invocation adds seconds, not
minutes. Executor documents the choice in the SUMMARY.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Stopwatch test → MSW server | The MSW server runs in-process; intercepts undici-fetch calls; serves Phase 2+3 fixtures. No real WHOOP traffic per ADR-0006. |
| Stopwatch test → tmp filesystem | tmpHome + tmpRepo are mkdtempSync'd; cleanup is best-effort but reliable across vitest runs. Production paths.dbFile is NEVER touched because RECOVERY_LEDGER_HOME relocates the entire home directory. |
| CI workflow → repo source | The workflow checks out the PR's HEAD and runs the test in isolation. No secrets are needed (MSW serves fake credentials). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-T9 | Tampering | A future regression could make the stopwatch flaky (running close to budget) | mitigate | Pitfall 1 documents the failure modes; the workflow has `timeout-minutes: 30` as a hard upper bound (kills flaky runs rather than retrying). If the budget tightens (e.g., 15min), the failure mode becomes more sensitive — a future plan may need to revisit the `npm ci` cache strategy or move to Docker-image-based runners. |
| T-05-I10 | Information Disclosure | Stopwatch test sets `WHOOP_CLIENT_ID=test_client` in env | accept | Fake credential; MSW intercepts before any real fetch. Not sensitive. |
| T-05-D5 | DoS | CI workflow burns ~5-10 minutes × 2 platforms per qualifying PR | mitigate | Path filter (D-13) limits qualifying PRs to those that actually move the needle (package.json, src/cli/**, bootstrap.ts, migrations). Doc-only PRs do not trigger. |
| T-05-SC | Tampering | No new npm installs in this plan | accept | Test uses MSW (already in deps); workflow uses pinned action SHAs copied from ci.yml. |
