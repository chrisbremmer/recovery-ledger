TDD plan: ship the remaining Wave 1 pure-domain modules — the weekly pattern detector + the fixed catalogs that drive actions and decision prompts. Four load-bearing constraints:

1. **D-11 5-candidate lock with dropped-rationale comments** — `candidates.ts` ships the module-load constant, the runtime Set, and a doc-comment citing `rhr_delta_prior_day` (multicollinearity with HRV) + `respiratory_rate_anomaly_prior_day` (low power for rare events) so a future reader doesn't ask why.

2. **D-12 trailing-28 pattern-test window + D-17 trailing-7 week-summary window kept distinct** — `pattern.ts` consumes the trailing-28 arrays (the service in Plan 04-07 supplies them); the trailing-7 week_summary slot is assembled outside this module so the two windows stay separated end-to-end (in the result object, in the renderer rendering, in the test fixtures).

3. **D-18 multi-detection policy + D-34 pattern_confidence on the detected arm** — when multiple candidates clear FDR, the pattern detector picks the cleared candidate with the smallest `p_adjusted` and returns it as `pattern.factor`. The full ranked list (all 5 candidates with cleared/refused flags + p_adjusted) lives in `candidate_results` per ADR-0004 §Consequences (unranked candidates as context, not as recommendation). The detector applies `'weak'` when 14 ≤ N_scored < 20 and `'strong'` when N_scored ≥ 20 to the `pattern_confidence` slot on the `detected` arm.

4. **ADR-0005 + D-09 + D-23 + D-26 banned-word lint at module-load** — both catalogs (`ACTION_CATALOG` and `DECISION_PROMPT_CATALOG`) have a sibling `*.test.ts` that iterates every entry and asserts `containsBannedToneToken(entry.text).hit === false`. This is the source-level layer; Wave 3 Plan 04-09 adds the rendered-output layer.

Purpose: REV-06 (weekly review surfaces worst-days + runs 5 candidates) + REV-07 (BH FDR @ q=0.10 + typed positive output for absence) + REV-08 (banned tone words enforced) + DEC-04 (decision prompt slot) all land here.   ### pattern.ts (~120 LOC)
  - Exports `detectWeeklyPattern(input: { cycles: Cycle[]; recoveries: Recovery[]; sleeps: Sleep[]; workouts: Workout[]; baselines: Record<'hrv_rmssd_milli', BaselineStats>; ianaZone: string; }, q?: number): { pattern: WeeklyPattern; candidate_results: CandidateResult[]; worst_days: WorstDay[]; }`. `q` default = 0.10. The cycles/recoveries/sleeps/workouts arrays are ALREADY-WINDOWED to trailing-28 from reviewed_date by the SERVICE caller (Plan 04-07), per D-12 — pattern.ts never re-windows.

  Algorithm (RESEARCH §6 verbatim):
  1. Filter `cycles` to those with `score_state === 'SCORED'` AND `baseline_excluded === 0` AND `day_recovery_score !== null`. Call this `scoredCycles`. (The trailing-28 windowing is already applied upstream per D-12.)
  2. If `scoredCycles.length < 14` → return `{ pattern: { kind: 'no_pattern', reason: 'insufficient_window_days' }, candidate_results: [], worst_days: [] }`.
  3. Compute `nWorst = Math.max(2, Math.floor(scoredCycles.length / 4))`.
  4. Sort `scoredCycles` ascending by `day_recovery_score`, tie-break by `start` ASC (Pitfall 1 chronological tie-break).
  5. `worst_days = sortedAscending.slice(0, nWorst).map(c => ({ date: c.start.slice(0, 10), recovery_score: c.day_recovery_score }))`.
  6. Build worst-day-id set + other-day-id set.
  7. For each candidate in `CANDIDATE_FACTORS`:
     - Assemble `sampleWorst[]` and `sampleOther[]` per factor (helper functions per candidate type):
       - `sleep_duration_prior_night` — for each cycle in worst/other, find the sleep that ended in the cycle's calendar day; take its `duration_minutes`. Skip null.
       - `sleep_debt_3d_rolling` — for each cycle, sum (sleep_need − actual_duration) over the prior 3 cycles' sleeps. Skip when any of the prior 3 is missing.
       - `day_strain_prior_day` — prior cycle's `day_strain`. Skip null.
       - `workout_timing_late_evening` — count of workouts in the prior cycle's 18:00-23:59 user-local window (use `ianaZone`). Returns 0 if no workouts, integer count.
       - `hrv_delta_prior_day` — prior cycle's `hrv_rmssd_milli` Z-score vs the trailing-30 baseline (uses `input.baselines.hrv_rmssd_milli`). Skip when baseline.mad === 0 or HRV is null.
     - If `sampleWorst.length < 2` OR `sampleOther.length < 4`: `candidate_results.push({ factor, p_raw: Number.NaN, p_adjusted: Number.NaN, cleared: false, refused: true, refusal_reason: 'sample_too_small' })`.
     - Else: `const { U, p } = mannWhitney(sampleWorst, sampleOther); candidate_results.push({ factor, p_raw: p, p_adjusted: 0, cleared: false, refused: false })` (p_adjusted filled after BH).
  8. Collect `nonRefusedPs = candidate_results.filter(r => !r.refused).map(r => r.p_raw)`.
  9. If `nonRefusedPs.length === 0` → return `{ pattern: { kind: 'no_pattern', reason: 'all_candidates_refused' }, candidate_results, worst_days }`.
  10. Run `benjaminiHochberg(nonRefusedPs, q)` → `{ rejected, adjusted }`. Map back to non-refused `candidate_results` positions; set `cleared = rejected[i]`, `p_adjusted = adjusted[i]`.
  11. If no `candidate_results.cleared` is true → return `{ pattern: { kind: 'no_pattern', reason: 'no_factor_cleared_fdr' }, candidate_results, worst_days }`.
  12. Else: find the cleared candidate with the smallest `p_adjusted` (D-18 multi-detection policy — pattern reports the single candidate with smallest p_adjusted; the rest stay in candidate_results as unranked context per ADR-0004 §Consequences). Determine `direction`: compute `median(sampleWorst)` vs `median(sampleOther)` for that factor; if `worstMedian < otherMedian` → `'worst_days_had_lower'`, else `'worst_days_had_higher'`. Determine `pattern_confidence` per D-34: `scoredCycles.length >= 20 ? 'strong' : 'weak'`. Return `{ pattern: { kind: 'detected', factor, statistic: { U: candidateU, p_raw, p_adjusted }, direction, pattern_confidence }, candidate_results, worst_days }`.
