<objective>
Ship the review service orchestrators (`daily.ts` + `weekly.ts`) that compose every Wave 1 domain module into the typed `DailyReviewResult` + `WeeklyReviewResult` shapes. This is THE chassis layer for Wave 4 — once these are in place, every CLI command + MCP tool + MCP resource that surfaces a review is a ≤5-line shim over `services.getDailyReview` / `services.getWeeklyReview`.

Purpose: REV-01..REV-07 + DEC-04 all anchor here. REV-04 (data-freshness lead) is encoded by putting `data_status` first in the DailyReviewResult schema and at the front of the data flow. REV-05 (insufficient → states what's missing + declines) is encoded by the explicit `confidence.tier === 'insufficient'` path that sets anomalies/actions to []. REV-07 (BH FDR + typed positive output) flows through `detectWeeklyPattern` from Plan 04-05. DEC-04 (decision_prompt when none in prior week) lands here via `decisionsRepo.countSince()`.

This plan also ships ~10 fixture JSON files under `tests/fixtures/review/` (Wave 0 gaps per VALIDATION.md) — these are the contract test inputs Wave 3 + Wave 4 will reuse. Per ADR-0006, all tests are fixture-only via the in-memory-db helper.

Output: 4 service files + 4 sibling tests + 10 fixture JSONs + 1 small repo extension (`syncRunsRepo.latestFinished()`).
</objective>
