<objective>
Ship the opt-in `concurrent_writers_stress` doctor probe (D-02 #9). Mirrors the Phase 2 D-23 10-fork concurrent-load test pattern but with 4 workers writing to a tmp DB. Subprocess-skip-gated like `mcp_stdout_purity` AND `--stress`-flag-gated so it never runs from MCP and never runs from a default CLI doctor invocation. The probe exists to diagnose concurrent-writer contention (Pitfall 12) when a user suspects sync + ad-hoc decision adds racing.

Purpose: when something IS racing in production, the user runs `recovery-ledger doctor --stress` and gets a concrete pass/fail signal. The 4-worker pattern matches Phase 3 D-30's `busy_timeout=5000` primitive (each worker should serialize cleanly behind the immediate lock; SQLITE_BUSY escape means the timeout is being beaten).

Output: 1 probe file + 1 worker script + 1 test file. Plan 05-06 wires the probe into runDoctor()'s last slot (PROBE_NAMES[13]).
</objective>

| Threat ID | Category | Component | Disposition | Mitigation Plan |
