    ```
    ## Comment phrasing must not collide with grep gates

    **Observed:** Phases 1-4 have repeatedly hit the same micro-issue: a doc-comment uses the literal substring of a Gate's target regex (e.g., a comment block containing the word `console.log` to explain why the gate exists, or the literal `server.registerTool(` to document the discipline), tripping the gate at CI time.

    **Convention:** When writing a doc-comment that REFERENCES a grep-gate target, never inline the literal grep target. Instead:
    - For `console.*` references: use "direct stdout writes" or "the console API".
    - For `process.stdout.write` references: use "stdout output" or "the write API".
    - For `server.registerTool` / `registerResource` / `registerPrompt` references: use "the registration boundary" or "the wrapper".
    - For `drizzle-orm` references: use "the ORM".
    - For `fetch(` references: use "HTTP requests" or "the network boundary".
    - For OAuth-token-URL references outside `token-store.ts`: use "the refresh endpoint".

    **Why:** the grep gates are word-boundary literal checks; case-insensitive variants and underscore breaks (e.g., `respiratory_rate`) don't collide, but a deliberate doc reference to the exact substring does. The convention preserves explainability in comments without breaking CI.

    **Phases affected:** 1, 2, 3, 4 — observed in every phase's close commit. Recommend adding to `agent_docs/conventions.md` §Comments section as a hard rule after this entry.
    ```
